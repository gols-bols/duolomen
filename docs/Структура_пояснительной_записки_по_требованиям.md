# Структура пояснительной записки

## 2. Практическая часть

### 2.1 Проектирование архитектуры проекта

В рамках практической части была спроектирована архитектура web-приложения для обработки заявок внутри образовательной организации. В качестве архитектурной основы выбран шаблон MVC, реализуемый средствами Laravel. Такой подход позволил разделить проект на несколько логических уровней: маршрутизацию, контроллеры, модели, представления и базу данных.

На уровне клиентского взаимодействия пользователь работает через браузерный интерфейс и через API, предназначенный для тестирования и демонстрации работы системы в Postman. Запросы от клиента передаются на серверную часть Laravel, где проходят маршрутизацию, проверку прав доступа, валидацию данных и обработку прикладной логики. После этого данные сохраняются в базе MySQL и возвращаются пользователю в виде HTML-страниц или JSON-ответов.

С точки зрения структуры данных ключевыми сущностями системы являются пользователи и заявки. Пользователь хранит сведения об имени, электронной почте, роли и закрепленном корпусе. Заявка содержит заголовок, описание, приоритет, статус, корпус, кабинет, имя заявителя, автора заявки и назначенного исполнителя. Такая модель данных позволяет реализовать прикладной сценарий эксплуатации системы в учебной организации.

В пояснительной записке целесообразно разместить не менее трех диаграмм:
- диаграмму общей архитектуры системы;
- диаграмму взаимодействия клиента, сервера и базы данных;
- схему данных или ER-диаграмму базы данных.

Пример описания диаграммы архитектуры: пользователь взаимодействует с web-интерфейсом или Postman, далее запрос поступает в Laravel-приложение, которое обрабатывает его через маршруты, контроллеры и модели, после чего обращается к MySQL и возвращает результат.

### 2.2 Разработка интерфейсов

Разработка интерфейсов выполнялась с учетом ролей пользователей и специфики предметной области. Основной задачей было создать понятную и визуально собранную систему, в которой пользователь может быстро выполнить свое действие без лишних шагов, а сотрудник системы получает удобный рабочий экран для обработки обращений.

В проекте были разработаны основные интерфейсные экраны: страница авторизации, список заявок, форма создания обращения, карточка отдельной заявки и форма обработки заявки сотрудником системы. Отдельное внимание было уделено отображению прикладных данных. Вместо формального показа только идентификатора заявки в интерфейсе выводятся имя заявителя, корпус, кабинет, приоритет, текущий статус и назначенный исполнитель.

Интерфейсы различаются в зависимости от роли пользователя. Обычный пользователь имеет доступ только к созданию и просмотру собственных заявок. Менеджер корпуса работает только с заявками своего корпуса. Администратор получает полный доступ ко всем обращениям и может назначать исполнителей, изменять статусы и контролировать поток заявок по всем корпусам.

Визуальная часть интерфейсов была доработана таким образом, чтобы система выглядела как законченный внутренний сервис, а не как набор шаблонных учебных форм. Для этого были переработаны карточки заявок, цветовые акценты статусов, сводные информационные блоки и композиция рабочих экранов.

### 2.3 Разработка клиентской части информационной системы

Клиентская часть проекта реализована на основе Blade-шаблонов Laravel. Такой подход позволил сформировать web-интерфейс без вынесения фронтенда в отдельное приложение, сохранив при этом понятную и управляемую структуру проекта. Пользователь взаимодействует с системой через браузер, а сервер динамически формирует страницы на основе полученных данных.

На клиентской стороне реализованы основные сценарии работы: вход в систему, просмотр списка доступных заявок, создание нового обращения, просмотр карточки заявки и, для сотрудников системы, редактирование параметров обращения. Для улучшения восприятия информации были реализованы цветовые статусы, фильтры по корпусу, приоритету и состоянию заявки, а также логичная визуальная группировка данных.

Дополнительно клиентская часть была адаптирована под демонстрационные задачи защиты проекта. В интерфейсе были устранены пустые и неинформативные элементы, добавлены поясняющие блоки, улучшено отображение ролей и зон ответственности. Благодаря этому клиентская часть стала не только функциональной, но и презентационной.

### 2.4 Разработка серверной части информационной системы

Серверная часть проекта разработана на языке PHP с использованием фреймворка Laravel. Она обеспечивает обработку запросов, разграничение прав доступа, валидацию входных данных, взаимодействие с базой данных и возврат результата пользователю. Структура серверной части включает маршруты, контроллеры, модели и middleware.

На стороне сервера реализована ролевая модель с тремя основными типами пользователей: `user`, `manager` и `admin`. Пользователь может создавать заявки и просматривать только собственные обращения. Менеджер корпуса имеет право работать только с заявками своего корпуса. Администратор управляет системой целиком и может изменять все ключевые параметры заявки.

Для хранения данных используется MySQL. Работа со схемой базы организована через миграции Laravel, что позволяет воспроизводить структуру таблиц на любом компьютере. Для заполнения базы демонстрационными данными используются сидеры, создающие тестовые учетные записи и примерные заявки.

Отдельным направлением разработки стала реализация API для тестирования системы через Postman. Были созданы JSON-маршруты для авторизации, получения списка заявок, просмотра отдельной заявки, создания новой записи и обновления существующей заявки. Для защищенных маршрутов используется Bearer-токен, что делает API удобным для демонстрации на защите.

### 2.5 Тестирование информационной системы

Тестирование проекта выполнялось на нескольких уровнях. Во-первых, проводилась проверка базовых пользовательских сценариев через web-интерфейс: вход в систему, создание заявки, просмотр карточки обращения, изменение статуса и фильтрация заявок. Во-вторых, тестировалась серверная часть и API через Postman.

При тестировании API были проверены как корректные, так и ошибочные сценарии. Для положительных сценариев проверялась успешная авторизация, получение списка заявок, создание новой записи и изменение статуса. Для отрицательных сценариев проверялась реакция системы на передачу неполных или некорректных данных, а также попытки обращения без токена или без достаточных прав доступа.

Результаты тестирования подтвердили работоспособность системы и корректность прикладной логики. Проверка базы данных после выполнения API-запросов позволила убедиться, что изменения действительно сохраняются в MySQL. На основании проведенного тестирования можно сделать вывод о том, что разработанная система соответствует поставленным задачам учебного проекта.

## 3. Экономическое обоснование

### 3.1 Определение затрат на проектирование

Экономическое обоснование проекта включает оценку трудовых и организационных затрат, необходимых для его разработки. Поскольку проект создавался в учебных условиях, в структуру затрат целесообразно включать прежде всего затраты времени разработчика, затраты на использование компьютерной техники и условные затраты на программные средства.

В состав затрат можно включить:
- анализ предметной области и формирование требований;
- проектирование базы данных и архитектуры приложения;
- разработку интерфейсов;
- реализацию клиентской и серверной части;
- тестирование и исправление ошибок;
- подготовку документации, презентации и отчетных материалов.

При расчете затрат на проектирование обычно определяется количество часов, затраченных на каждый этап, после чего производится пересчет в условную стоимость разработки. Такой расчет показывает, что даже учебный проект требует комплексных трудозатрат и включает не только программирование, но и проектирование, тестирование и оформление результата.

### 3.2 Расчет экономической эффективности

Экономическая эффективность проекта в учебной работе может быть рассмотрена через сокращение времени обработки обращений, уменьшение потерь заявок и повышение прозрачности работы службы технической поддержки. Внедрение информационной системы позволяет перевести процесс регистрации заявок из неформального режима в структурированный цифровой формат.

К основным положительным эффектам относятся:
- сокращение времени регистрации и поиска обращения;
- снижение вероятности потери заявки;
- упрощение контроля статусов и исполнителей;
- повышение прозрачности распределения нагрузки;
- накопление истории обращений для последующего анализа.

Даже если на этапе курсового проекта система используется как учебный прототип, ее применение уже демонстрирует организационную и потенциальную экономическую полезность. В перспективе дальнейшее развитие проекта может привести к снижению трудозатрат сотрудников на учет обращений и повышению эффективности работы технической поддержки.

## Заключение

В ходе выполнения проекта была разработана информационная система для управления заявками, предназначенная для использования внутри образовательной организации. В работе были последовательно решены задачи проектирования архитектуры, построения базы данных, разработки интерфейсов, реализации клиентской и серверной части, а также тестирования системы.

Практическим результатом работы стал функционирующий web-проект на базе Laravel и MySQL с разграничением ролей пользователей, делением по корпусам, системой обработки заявок и отдельным API для тестирования в Postman. Разработанная система может использоваться как учебный пример прикладной web-разработки и как основа для дальнейшего развития более полного внутреннего сервиса.

Полученные результаты подтверждают достижение поставленной цели. Проект продемонстрировал, что даже в рамках учебной разработки возможно создать логически завершенную и технически состоятельную систему, объединяющую проектирование, программирование, тестирование и документирование результата.
